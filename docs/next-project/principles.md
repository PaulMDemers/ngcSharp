# Principles

## Build A Diagnostics Platform First

An emulator without observability is a slow guessing machine. The first usable version should not aim to play games. It should load binaries, execute bounded instruction budgets, dump structured run summaries, expose device state, and make failures repeatable.

Useful early outputs:

- Final PC, stop reason, instruction count, selected registers.
- Device counters such as FIFO bytes, interrupts raised, DMA transfers, media reads, controller polls.
- Profile summaries: top PCs, top callers, branch targets, hot loops.
- Optional traces with hard caps so runs cannot flood the workspace.
- Machine-readable JSON plus compact CSV summaries.

Human console logs are helpful while exploring, but they should not be the primary artifact. Every recurring question should eventually become a field in a summary file.

## Work In Bounded Runs

Long-running emulation hides causality. Prefer bounded probes:

- Stop after a fixed instruction/cycle budget.
- Stop on a PC, write, load, call target, FIFO offset, interrupt, or hot-PC threshold.
- Use wall-clock watchdogs around benchmark scripts.
- Capture enough state to resume the investigation without rerunning immediately.

Bounded runs turn compatibility into a sequence of checkpoints. A run that stops at the same PC with the same counters is a stable milestone, even if the emulator is still far from playable.

## Prefer Small Legal Fixtures

Retail software is useful as a stress test, but it is a terrible first repro. Build a ladder:

1. Synthetic binaries generated by the repo.
2. Tiny source-built homebrew fixtures.
3. Public demos and utilities with clear redistribution rules.
4. User-provided commercial images, kept out of git.

Each fixture should test one claim when possible: one instruction family, one DMA path, one timer interrupt, one texture format, one blending mode.

## Make State Explicit

Global hidden state is the enemy of save states, deterministic tests, and debugging. Keep CPU, memory, devices, scheduler, and media state explicit. Even when a monolithic bus is convenient early, design it so devices can later be split into smaller testable units.

If a subsystem needs time, give it an explicit clock. If it raises interrupts, expose why and when. If it mutates memory, make that visible to trace or watch tooling.

## Optimize Only After You Can Measure

Interpreter speed will eventually hurt. Do not optimize blind. Add timings and counters first, then optimize the largest measured cost.

Good optimization evidence:

- Before/after run summaries.
- Same stop reason and compatibility counters.
- Same image hash or intentional visual difference.
- Timing breakdown by phase, not just total wall time.

## Be Honest About Shortcuts

Compatibility fast-forwards and high-level emulation can be useful, but they must not become a junk drawer for missing hardware. A safe shortcut is exact, narrow, observable, and covered by tests. A risky shortcut changes game state because "that gets farther."

When in doubt, implement the lower-level behavior or add diagnostics before skipping code.

## Documentation Is Part Of The Emulator

A running log is not overhead. It is the memory of why a weird helper exists, which artifact proved it, and what not to try next. Write down:

- The observed code pattern or hardware behavior.
- The artifact path or command that proved it.
- The current blocker and why it is believed to be the blocker.
- The next experiment.

Future you is another developer. Be kind to that person.
