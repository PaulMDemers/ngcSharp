# Test Assets And Fixtures

## Build A Test Ladder

Compatibility improves fastest when tests progress from tiny and controlled to broad and chaotic.

Recommended ladder:

1. Unit tests for CPU instructions, memory, and devices.
2. Synthetic executable generated by the repo.
3. Source-built homebrew fixtures.
4. Public demo/application binaries with clear redistribution rules.
5. Local user-provided retail media.
6. Reference comparisons against a mature emulator or real hardware captures.

Do not skip the lower rungs. Retail failures are too compound: CPU, scheduler, media, input, storage, graphics, audio, and timing can all fail at once.

## Synthetic Executables

Synthetic binaries are excellent for first boot and regression coverage. They should be tiny enough that every instruction and memory write is intentional.

Useful synthetic tests:

- Load sections at known addresses.
- Write a magic value to memory.
- Hit a terminal self-branch or explicit halt.
- Exercise branch/link behavior.
- Exercise stack setup and ABI basics.
- Write a tiny framebuffer.

The goal is to validate the loader and CPU loop without depending on an SDK.

## Source-Built Homebrew

Homebrew fixtures are the bridge between unit tests and real software. Build them from source in the repo when license-compatible.

Good fixture traits:

- One subsystem focus.
- Deterministic output.
- Small binary.
- Minimal dynamic allocation.
- Fixed framebuffer or command stream.
- Assertions through memory writes, hashes, or frame pixels.

Examples by subsystem:

- Input: poll controller, write button state to memory or screen.
- Storage: read a known sector/file and print status.
- Timer/interrupt: count callbacks over a bounded frame window.
- Graphics: draw color bars, texture formats, blend modes, copy operations.
- Audio: fill a small buffer and expose counters, even before sound output works.

## Public Demo Sweeps

Once source-built fixtures pass, collect a local inventory of public demos and utilities. Do not turn every file into a committed target. Use an inventory script to discover candidates, then promote only the useful ones into a curated matrix.

Promotion criteria:

- It exercises a subsystem not covered by smaller fixtures.
- It has a stable command and result.
- It has clear provenance.
- Its output can be summarized without large committed artifacts.
- It fails in a way that teaches the emulator something.

## User-Provided Retail Benchmarks

Retail benchmarks are best used as named local scenarios:

- Short boot smoke target.
- Longer CPU/resource target.
- Graphics capture target.
- Deep diagnostic target with expensive traces.

Each target should define:

- Instruction/cycle budget.
- Watchdog timeout.
- Input setup.
- Storage/media setup.
- Expected stop reason or milestone.
- Summary fields to track.

Avoid "run until it plays." A vague benchmark is not a debugging tool.

## Artifacts And Reproducibility

Every fixture run should write to a timestamped artifact directory:

- `stdout.txt` and `stderr.txt`.
- `run-summary.json`.
- Optional traces.
- Optional images.
- Optional per-device CSVs.
- Suite-level `summary.csv` and `summary.json`.

The command should be recoverable from the artifact or matrix. If a result cannot be reproduced, it is anecdote, not evidence.

## Hashes Are Useful But Brittle

Image/frame hashes are powerful for small deterministic fixtures. They are risky for broad retail captures where timing, scene selection, or raster budget may change legitimately.

Use hashes for:

- Tiny homebrew frames.
- Known command-stream fixtures.
- Synthetic framebuffer tests.

Prefer structural expectations for broad runs:

- Nonblack pixel count.
- Draw count.
- FIFO bytes.
- Copy count.
- Selected frame source.
- Top PC/caller profile.
